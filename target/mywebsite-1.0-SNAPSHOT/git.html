üîç What is Version Control?

Definition: Version Control is a system that records changes to files over time so you can recall specific versions later.

Use Case: Used in software development to collaborate, track changes, undo errors, and manage code across environments.

Key Features:

  • Track history of code files
  • Collaborate across teams
  • Revert to previous versions
  • Handle conflict resolution

üìÇ Types of Version Control Systems

1. Local Version Control

Each user stores file versions on their local machine using tools like RCS. Simple but not ideal for collaboration.

2. Centralized Version Control System (CVCS)

Definition: All files and version history are stored in a central server. Users pull/push changes to/from that server.

Examples: Subversion (SVN), Perforce, CVS


Pros:
- Easy to manage for small teams
- Centralized backup

Cons:
- Single point of failure
- Limited offline access

3. Distributed Version Control System (DVCS)

Definition: Every user has a full copy of the repository including history. Changes are shared via pull/push.

Examples: Git, Mercurial, Bazaar


Pros:
- Full history available offline
- Faster operations
- Branching and merging is easier

Cons:
- Slightly higher learning curve

üêô What is Git?

Git is a free, open-source, distributed version control system created by Linus Torvalds in 2005.

It is designed for speed, efficiency, and collaboration, especially in large-scale software development.

Why Git?

  • Distributed architecture (no central server needed)
  • Lightweight branching and merging
  • Efficient file tracking and snapshots
  • Massive community support and tooling (GitHub, GitLab, Bitbucket)

Git Guide - DevOps World

1. Configuring Git

Purpose: Define who you are in the Git environment and control how Git behaves for you.

Use Case: Essential before you start committing to ensure your identity is linked to your work and to set up preferences like your editor or merge tool.

git --version
git config --global user.name "Your Name"
git config --global user.email "your@email.com"
git config --local user.name "Your Name"
git config --local user.email "your@email.com"
git config --list
git config --global --remove-section user
git config --local --remove-section user

2. Initializing a Git Repository

Purpose: Begin version control in a directory so Git can start tracking changes.

Use Case: Use this when you're starting a new project or want to enable Git tracking in an existing project folder.

git init
ls -a
git status

3. Adding Changes to Staging

Purpose: Move files from your working directory into the staging area for the next commit.

Use Case: Stage specific or all changes before committing to keep your commits clean and organized.

git add filename
git add .
git status

4. Committing Changes

Purpose: Save a snapshot of your staged changes to the local repository history.

Use Case: After staging your changes, committing helps you keep track of progress and collaborate effectively.

git commit -m "Commit message"
git commit
git commit -am "Commit message"
git commit --amend

5. Viewing Commit History

Purpose: Review changes made over time, who made them, and how the project has evolved.

Use Case: Helpful when debugging, auditing changes, or understanding feature evolution and collaboration history.

git log
git log --oneline
git log --graph
git show <commit_hash>

6. Branching

Purpose: Create isolated environments to work on different features, bug fixes, or experiments without affecting main code.

Use Case: Essential for managing development workflows and keeping changes organized by task or feature.

git branch
git branch new_branch
git switch new_branch
git checkout new_branch
git checkout -b feature
git branch -m new_name
git branch -d feature
git branch -D feature

7. Merging

Purpose: Combine the contents of one branch into another, usually integrating features into main development.

Use Case: Use when a feature branch is complete and needs to be merged into main or another development branch.

git merge feature
git merge --abort
git rebase main
git rebase -i main

8. Stashing

Purpose: Temporarily store changes that you‚Äôre not ready to commit so you can switch contexts safely.

Use Case: When you're mid-task but need to quickly change branches or fix something urgent.

git stash
git stash list
git stash apply
git stash pop
git stash clear

9. Resetting Changes

Purpose: Revert changes in your working directory, staging area, or commit history depending on mode.

Use Case: Useful for undoing mistakes or cleaning up your commit history before pushing to remote.

git reset commit_hash
git reset --soft commit_hash
git reset --mixed commit_hash
git reset --hard commit_hash

10. Checking Differences

Purpose: Compare changes between working directory, staging area, and last commits to see what has changed.

Use Case: Ideal for reviewing before staging or committing, and also during conflict resolution in merges.

git diff
git diff --staged
git diff HEAD

11. Deleting Files

Purpose: Remove files from version control and optionally from the file system.

Use Case: Helps clean up the repository by removing unnecessary or outdated files while keeping history clean.

git rm filename
git rm --cached filename

12. Remote Repositories

Purpose: Connect your local repository to remotes like GitHub, GitLab, or Bitbucket to collaborate and store code in the cloud.

Use Case: Enables team collaboration, centralized backups, CI/CD integration, and easy code sharing across environments.

git remote add origin <url>
git push origin main
git pull origin main
git fetch
